# 2026-04-01

- Continued `P0 security` on `user_settings` instead of expanding new product surface:
  - kept the client contract unchanged (`GET /api/user-settings` for presence flags, `GET /api/user-settings/pdf-passwords` for the narrow secret path)
  - added a legacy-rotation path so old plaintext PDF passwords do not stay in the table indefinitely after the new encryption rollout
- `GET /api/user-settings/pdf-passwords` now performs opportunistic upgrade-on-read:
  - if a stored password field is already `v1.*` ciphertext, it is only decrypted for the response
  - if a stored password field is legacy plaintext, the Worker now encrypts it with `USER_SETTINGS_SECRET_KEY` and upserts the ciphertext back into `user_settings`
  - the response still returns the raw password string to the caller so the Gmail sync flow remains unchanged
- This keeps the rollout backward-compatible while shrinking the plaintext tail:
  - no explicit admin migration command is required for active users
  - rows that are never read through the secret path may still remain plaintext until a later dedicated migration/rotation pass
- Added regression coverage in `apps/api/tests/auth-accounts.test.ts` for the upgrade-on-read behavior so legacy plaintext rows are re-written as ciphertext after the first explicit secret fetch.
- Started `P0 schema discipline` so database evolution has history instead of a single mutable baseline:
  - added `supabase/migrations/20260401000000_baseline.sql` as the first committed schema-history entry
  - documented `supabase/schema.sql` as the fresh-bootstrap snapshot and `supabase/migrations/` as canonical change history
  - updated repo governance/docs so future schema changes must update both the migration history and the latest bootstrap snapshot
