# 2026-03-26

- Added a dedicated `taishin_pdf_password` setting across the local stack:
  - extended `user_settings` schema with a Taishin-specific PDF password field
  - exposed the new field from the `user-settings` API
  - added a Taishin password input in the settings panel
  - Gmail PDF sync now prefers `taishin_pdf_password` for Taishin statements before falling back to the default PDF password
- User rule captured for current product behavior:
  - Taishin credit-card PDF passwords can differ from the default bank password pattern
  - local UI should allow an explicit Taishin override so Gmail import does not fail with `Incorrect Password`
- Hardened the user-settings failure path after validation:
  - the web app had been treating non-2xx `user-settings` responses as success
  - this could make the settings panel appear to save successfully even when the backing schema did not yet contain `taishin_pdf_password`
  - Gmail sync now fails early with an explicit missing-password / missing-schema message instead of falling through to PDF.js `No password given`
- Expanded the Taishin PDF parser for a newly validated statement layout:
  - Taishin rows can use ROC full dates like `115/02/14` instead of only `MM/DD`
  - local statements can place a trailing country code such as `TW` after the booked amount
  - parser should keep using the booked TWD amount and ignore the trailing country/location column
- Expanded the CTBC PDF parser for a newly validated statement layout:
  - CTBC rows can use ROC full dates like `114/12/20`
  - CTBC statement rows place the booked TWD amount before the card last-4 column
  - rows may include trailing country / currency / foreign-amount columns after the card suffix, which should not block transaction parsing
- Improved Gmail PDF import diagnostics and text extraction:
  - the frontend now reports which bank parser failed and includes a short preview of the extracted text instead of a single generic parse-failure message
  - PDF text extraction now sorts text fragments left-to-right within the same visual row before joining them, reducing parser failures caused by column order scramble from PDF.js
- Further hardened PDF row reconstruction after CTBC validation:
  - the extractor now uses x-gap-aware joining instead of blindly inserting spaces between every fragment in a row
  - punctuation-heavy fragments such as `/`, `:`, `-`, and adjacent date tokens should stay attached during reconstruction
  - parse-failure previews now prioritize lines that look transaction-like (date/card-number bearing) instead of only the first visible header rows
- Added a second-layer CTBC recovery path for badly fragmented text extraction:
  - date-like tokens with spaces around `/` or split separators are normalized before bank parsing
  - CTBC parsing now falls back to scanning the full normalized text stream when transaction rows are not preserved as clean line breaks
- Replaced fixed Y-rounding with tolerance-based row clustering in PDF extraction:
  - text fragments are now assigned to the nearest visual row within a Y tolerance instead of using a single rounded Y bucket
  - this should recover statement rows whose columns land on slightly different text baselines and were previously split into multiple partial lines
